# ГЛОБАЛЬНЫЙ СТАНДАРТ ИМПЛЕМЕНТАЦИИ

> **Универсальный стандарт для агентов разработки и имплементации кода**

## Связь с другими стандартами

Данный стандарт - **высокоуровневые принципы**. Детали в:
- @docs/standards/tdd-standard.md - TDD процесс

## Базовые принципы разработки (ВЫСШИЙ ПРИОРИТЕТ)

**КРИТИЧНО:** Эти принципы имеют высший приоритет над детальными практиками.

### Краткие основы:

1. **Писать элегантный код, который решает задачу**
2. **Не добавлять обратную совместимость, если не запрошено явно**
3. **После каждого блока кода:** линтинг → компиляция → тесты → запуск
4. **Код должен быть:** чистым, читаемым, DRY, предпочитать правку вместо добавления
5. **Избегать:** ненужных откатов, избыточного версионирования, чрезмерного тестирования
6. **Качество превыше скорости** - лучше потратить время и написать качественно
7. **При неуверенности - спрашивать** с озвучиванием рекомендаций

## Подход к разработке

### Процесс с пользователем

1. **Обсуждение** → исследовать варианты → уточняющие вопросы
2. **ТЗ** → сформулировать → разбить на задачи → получить подтверждение
3. **Реализация** → спросить **"Приступаем к реализации?"** → дождаться подтверждения

## Принципы простоты

### Метрики (ОБЯЗАТЕЛЬНО):
- Cyclomatic Complexity < 10
- Function length < 30 строк
- Class length < 200 строк
- Parameters < 5 (объект для >5)
- Nesting < 4 уровней

### Правило 3-х альтернатив:
1. Придумай 3 решения
2. Выбери простейшее работающее
3. Избегай первого что пришло в голову

### Code Review вопросы (ВСЕГДА):
- "Можно ли это сделать проще?"
- "Зачем эта сложность?" (функция >30 строк)
- "Есть ли стандартное решение?"
- "Понятно ли junior разработчику?"

### Стоп-сигналы сложности 🚨:
- "Это очень умное решение" 🚩
- "Только я понимаю" 🚩
- "Потом разберёмся" 🚩
- "Тут нужно быть осторожным" 🚩
- Коллега спросил "что это?" 2+ раза 🚩

### Анти-паттерны:
- Преждевременная абстракция
- Generic hell: `T<K<V<U>>>`
- God функции (>3 разных вещи)
- Callback ад (>3 уровней)
- Магические числа без объяснения

### Триггеры рефакторинга:
- Дебаг >30 мин = упростить
- Тест сложнее кода = упростить
- Боишься менять = упростить архитектуру
- Повторяешь объяснение = комментарий или упростить

## TDD подход

**Детали в:** @docs/standards/tdd-standard.md

### Цикл (ОБЯЗАТЕЛЬНО):
1. **RED** - тест ПЕРВЫМ → должен упасть
2. **GREEN** - минимальный код → тест проходит
3. **REFACTOR** - улучшить → тесты зеленые

### КРИТИЧНО:
- ❌ **ЗАПРЕЩЕНО** изменять тесты для устранения ошибок компиляции
- ✅ **ВСЕГДА** изменяйте код под требования теста

## Работа с зависимостями

- **Выбор:** популярные, активные, совместимые, минималистичные
- **Управление:** фиксированные версии, регулярные обновления, security проверки

## Архитектурные принципы

- **Файлы:** одна ответственность, согласованное именование, логическая группировка
- **Паттерны:** SOLID, разделение слоев, DI для слабой связанности

## Качество

### Code Review:
- Самопроверка перед отправкой
- Соответствие стандартам
- Учёт производительности

### Рефакторинг:
- Постоянное улучшение
- С покрытием тестами
- Мелкими шагами

## Обработка ошибок

- **Принципы:** Fail Fast, Graceful Degradation, информативные сообщения
- **Практики:** логирование, валидация входных, обработка всех типов

## Производительность

- **Оптимизация:** профилирование → узкие места → баланс с читаемостью
- **Мониторинг:** метрики, алерты, capacity planning

## Безопасность

- **Принципы:** валидация/санитизация всех входных, auth/authz, HTTPS везде
- **Практики:** секреты в env, минимальные привилегии, регулярные обновления
